# EVA 前台界面重构设计

> 日期：2026-04-08
> 范围：主窗口、设置、约定相关前台体验、增殖窗口相关前台体验的统一重构
> 目标参考：Cherry Studio 风格的现代化桌面应用体验，但保留 EVA 的多能力桌面工具属性与 Win7 兼容策略

---

## 1. 背景与目标

当前 EVA 的整体程序结构、能力闭环和模块拆分已经较成熟，尤其 `Widget` / `Core` / `Service` 这一层次分工清楚，功能完整。但前台界面仍然偏“功能集中式控制台”风格：入口密集、信息层级不够清晰、视觉语言偏旧、页面容器和能力分区之间缺乏统一产品化设计。

本次重构不聚焦推理后端，不改变 EVA 的核心能力模型，而是聚焦前台体验升级：

- 以聊天与模型切换为默认主任务。
- 将工程能力（skills / MCP / Docker / 工作目录 / 终端）从首屏堆叠控件重构为按需展开的上下文工具层。
- 将知识库、媒体、设置等能力改造成清晰独立的现代页面。
- 用统一的布局体系、页面容器、卡片语言、QSS 设计语言与导航结构替代现有零散控件组织方式。
- 在主方案现代化的前提下，为 Win7 提供视觉与效果上的降级路径，而不破坏整体产品结构。

---

## 2. 设计原则

### 2.1 产品原则

1. **聊天优先**：默认首页与默认注意力中心必须是聊天与模型切换。
2. **能力不丢失**：可以大胆重排路径，但不删除 EVA 现有关键能力。
3. **结构优先于装饰**：现代感主要来自信息架构、留白、层级、卡片与状态组织，而不是复杂特效。
4. **主任务清晰，辅助能力后置**：高频任务始终前置，工程/知识库/媒体等复杂能力采用层级化展开。
5. **同一产品，不是拼装面板**：主窗口、设置、增殖窗口等都应服从同一套导航与视觉系统。

### 2.2 技术原则

1. **优先复用现有 Qt Widgets / QSS 体系**，避免为追求视觉效果引入高风险 UI 技术。
2. **UI 重构与底层逻辑解耦**，尽量不改动 `core/session`、`core/toolflow`、`service/backend` 的职责边界。
3. **页面容器化**：把现有散落在 `Widget` / `Expend` 中的功能区抽象为更清晰的页面与侧栏容器。
4. **可分阶段落地**：允许先建立新的导航和页面骨架，再逐步迁移旧能力视图。
5. **Win7 可降级**：主结构统一，视觉质感可以按平台降级。

---

## 3. 总体信息架构

新版 EVA 前台采用“现代桌面工作台”结构，而不再以单窗口大面板形式组织。

### 3.1 一级架构

一级导航固定为：

- 聊天
- 工程
- 知识库
- 媒体
- 设置

### 3.2 空间结构

整体窗口划分为四层：

1. **最左窄侧栏**：一级导航，只承载全局入口与应用级切换。
2. **左主侧栏**：当前页面的二级内容，如会话列表、设置目录、知识库列表等。
3. **右主内容区**：当前任务的主画布，如聊天线程、知识库详情、媒体工作区、设置详情。
4. **右侧抽屉/上下文面板**：按需展开的辅助能力层，主要承载工程相关上下文（skills / MCP / Docker / 工作目录 / 终端等）。

### 3.3 与现有架构的关系

- 现有 `Widget` 仍是主窗口入口与状态机中枢，见 `src/widget/widget.cpp`。
- 现有 `Expend` 里的知识库、媒体等能力应逐步被重新映射为新的页面容器，而不是继续维持独立割裂的旧式功能窗。
- 现有主题、颜色与输出渲染逻辑（如 `src/widget/widget_settings.cpp`、`src/widget/widget_output.cpp`）可作为迁移基础，但需要被统一到新的视觉系统下。

## 4. 聊天页设计

聊天页是新版 EVA 的默认首页与核心工作台。

### 4.1 页面定位

打开 EVA 后，用户第一眼应感知到：

- 当前可以立即开始对话。
- 模型切换是显性能力。
- EVA 是桌面 AI 工作台，而不是参数控制台。

### 4.2 布局结构

#### 左栏

左栏承载与对话直接相关的导航和列表：

- 新建会话入口
- 全局搜索
- 当前模型快速切换
- 最近会话
- 固定会话
- 项目会话
- 运行状态摘要（当前模式 / 当前模型 / 工程能力挂载状态）

#### 右侧主内容区

右侧主内容区承载完整聊天体验：

- 顶部会话工具栏：会话标题、模型切换、上下文长度/速度/KV 等状态胶囊、重置/停止/工程抽屉按钮
- 中部消息流：用户、助手、tool、think 的统一消息渲染层
- 底部输入区：现代编辑器式输入框，整合上传、语音、发送、轻量状态提示

### 4.3 首页启动面板

在“尚未进入具体会话”时，聊天页显示轻量启动面板，而不是空白区：

- 常用模型
- 最近会话
- 最近项目/工作目录
- 最近知识库入口
- 最近媒体任务
- 恢复工程环境入口

### 4.4 消息视觉策略

重新定义消息层级：

- **用户消息**：清晰、简洁、次强调背景
- **助手消息**：正文优先，适合长文阅读
- **Tool 消息**：改成结构化消息块，弱化“日志堆叠感”
- **Think 消息**：视觉上弱于正文，强调辅助性

重点不是增加更多颜色，而是用留白、层级、标题、边框、字体权重来建立区分。

## 5. 工程页与右侧抽屉设计

### 5.1 页面定位

工程页是项目工作流中心，而不是杂项配置堆场。

它服务于：

- 工作目录管理
- Docker 沙盒环境
- skills
- MCP
- 工程终端
- 工程环境摘要

### 5.2 工程页主视图

工程页主内容区保持克制，只展示：

1. **项目头部卡**
   - 当前工作目录
   - 环境状态
   - Docker 状态
   - 启用的 skills / MCP 数量
   - 最近工程操作摘要

2. **工程动作区**
   - 打开工作目录
   - 切换工作目录
   - 启用/关闭工程模式
   - 快速进入终端
   - 快速打开 skills / MCP 抽屉

3. **最近工程上下文**
   - 最近项目
   - 最近 skill 组合
   - 最近 MCP 配置
   - 最近终端任务/命令摘要

### 5.3 右侧抽屉

工程细节配置与操作全部进入右侧抽屉，采用分段或 tabs 组织：

- **环境**：工作目录、Python/Node/Compile 探测、Docker 模式、镜像/容器状态
- **Skills**：技能列表、导入、启停、删除
- **MCP**：MCP 服务状态、工具列表、刷新/断开/配置入口
- **终端**：工程终端输出与命令交互

### 5.4 与现有实现的映射

- `SkillManager::composePromptBlock` 与技能管理逻辑继续复用，见 `src/skill/skill_manager.cpp`。
- Docker 目录与技能挂载逻辑继续复用，见 `src/widget/widget_engineer.cpp`、`src/utils/docker_sandbox.cpp`。
- MCP 工具与服务状态继续复用现有 `xMcp` / `Expend` 相关逻辑，但 UI 承载方式迁移到新的工程抽屉。

### 5.5 交互决策

- 聊天页也允许直接打开工程抽屉，不强制切页。
- 工程页负责总览与快捷动作。
- 右侧抽屉负责配置与执行细节。
- 终端默认折叠，只有在用户主动展开或运行工程相关命令时自动显性化。

## 6. 知识库页设计

### 6.1 页面定位

知识库页被定义为“资源库产品页”，不再只是一个技术设置面板。

### 6.2 页面结构

- **左侧子栏**：全部知识库、最近使用、收藏、导入记录
- **右侧主区顶部**：搜索、筛选、导入入口、新建入口
- **右侧主区中部**：卡片/表格混合视图的知识库列表
- **详情区**：索引状态、嵌入状态、更新时间、条目规模等

### 6.3 首页关系

知识库不会在首页大面积展开，只显示为聊天首页中的“最近知识库”卡片入口。用户点击后进入完整知识库页。

### 6.4 体验原则

- 把知识库当作资源对象管理
- 强调状态可见性与操作明确性
- 导入流程尽量向向导式体验靠拢
- 避免大量技术参数首屏暴露

---

## 7. 媒体页设计

### 7.1 页面定位

媒体页统一承载图像、音频与视觉相关能力，是创作型工作区。

### 7.2 页面组织

顶部使用任务类型切换：

- 图像
- 音频
- 视觉输入

主区域采用：

- 左侧参数区
- 右侧结果预览区
- 下方或侧边任务历史/日志抽屉

### 7.3 体验原则

- 结果优先于参数
- 参数组织模块化，减少旧式表单堆砌感
- 日志与历史默认不抢主视觉中心
- 保持与聊天页一致的设计语言，但允许媒体页拥有更明显的创作工具属性

---

## 8. 设置页设计

### 8.1 页面定位

设置页是现代设置中心，不再是长滚动表单集合。

### 8.2 结构

- **左侧目录**：通用 / 外观 / 模型 / 本地后端 / 链接模式 / 工程 / Skills / MCP / 媒体 / 高级
- **顶部搜索**：关键字定位设置项
- **右侧详情**：卡片化 section 布局

### 8.3 交互策略

- 常用设置置顶
- 需要重启/重载的设置明确提示
- 危险操作单独分组
- Win7 兼容项单独归类并明确说明降级效果

### 8.4 与现有实现的映射

- 现有全局主题、字体、输出字体与配色计算逻辑可在 `src/widget/widget_settings.cpp` 基础上重构。
- 现有 `settings_dialog` 不再作为旧式独立参数板，而应转换为新设置中心页面容器的一部分。

---

## 9. 视觉系统设计

### 9.1 总体风格

采用 **极简浅色生产力桌面 UI**：

- 主背景：暖白或浅灰白
- 卡片背景：纯白
- 分割线：极浅灰
- 主强调色：低饱和蓝
- 次强调色：蓝灰、石墨灰

目标是获得 Cherry Studio 式现代桌面感，但不直接复制其视觉，而是形成 EVA 自己的简洁生产力风格。

### 9.2 控件语言

- 统一中小圆角
- 更轻的描边与边界
- 更大的留白与更明确的区域分组
- 按钮层级为：主按钮 / 轻按钮 / 文本按钮
- 图标统一为现代线性图标

### 9.3 字体与密度

- 聊天阅读区：可读性优先，行高适中
- 列表区：略紧凑
- 设置区：中等密度
- 工具区：按信息量做紧凑布局

默认密度相较现状更现代、更松弛，但保留可切换的兼容紧凑布局能力。

### 9.4 主题体系

本次主方案为极简浅色主题。

后续允许保留多主题机制，但新主题系统必须先围绕一套统一 token 构建：

- 背景层级
- 文字层级
- 边框层级
- 状态色
- 间距、圆角、阴影 token

避免继续以零散 QSS 覆盖方式维护主题。

## 10. Win7 兼容策略

### 10.1 目标

在保持新版产品结构一致的前提下，为 Win7 提供视觉和效果层面的降级方案。

### 10.2 策略

主方案优先现代化，但必须满足以下兼容策略：

1. **同一套信息架构**：Win7 与新系统保持相同导航、页面与功能路径。
2. **效果降级，不做结构分叉**：Win7 允许减少阴影、透明度、动画与复杂绘制效果。
3. **避免高风险 UI 技术依赖**：优先使用 Qt Widgets + QSS + 稳定自绘实现现代观感。
4. **保留兼容密度与样式 fallback**：当环境较旧时切换到更稳妥的边框、间距、滚动条和消息块样式。

### 10.3 具体降级项

- 减少或取消复杂阴影
- 降低透明叠加与高频动画
- 提供更朴素的抽屉/卡片边框样式
- 对复杂绘制控件准备 fallback 皮肤
- 在高风险环境下使用更紧凑、边界更明确的控件样式

### 10.4 原则

用户在 Win7 上看到的应仍然是“同一个新版 EVA”，而不是旧版界面残留。

## 11. 组件与架构建议

### 11.1 新的 UI 分层建议

建议在现有 `src/widget` / `src/expend` 基础上引入新的页面容器分层：

- `AppShell`：整体窗口骨架（一级导航 + 左栏 + 主区 + 右抽屉）
- `ChatWorkspacePage`
- `EngineerWorkspacePage`
- `KnowledgeWorkspacePage`
- `MediaWorkspacePage`
- `SettingsWorkspacePage`
- `ContextDrawer`：统一右侧抽屉容器

这些可以先作为 Qt Widgets 组合类落地，不要求一开始就拆到独立进程或引入 QML。

### 11.2 迁移策略

迁移应分阶段：

1. 建立新的应用骨架与导航系统
2. 将聊天页迁移为新工作台
3. 将工程能力收拢到右侧抽屉
4. 将知识库/媒体重做为独立页面
5. 将设置页重做为设置中心
6. 最后统一主题 token 与 QSS 体系

### 11.3 不纳入本次范围

以下内容不属于本次设计主目标：

- 推理后端架构改造
- 工具协议与 MCP/Skill 后端协议重写
- Session / ToolFlow / BackendCoordinator 的逻辑职责重构
- 业务功能增加或减少

本次主要是 **前台重构与交互重编排**。

## 12. 测试与验收思路

### 12.1 验收目标

1. 用户打开应用后，首屏明确呈现聊天与模型切换。
2. 所有核心能力仍可达：工程、知识库、媒体、设置、skills、MCP、Docker、终端。
3. 新旧功能路径虽然变化，但整体学习成本下降。
4. 页面风格统一，不再出现明显割裂感。
5. Win7 下可以接受视觉降级，但结构和功能路径一致。

### 12.2 回归重点

- 主窗口状态切换
- 聊天发送、停止、重置、恢复历史
- 工程抽屉中 skills / MCP / Docker / terminal 的交互
- 设置项保存与即时刷新
- 知识库导入、索引、查询相关入口
- 媒体任务的参数配置与结果展示
- Win7 / 旧环境的 UI 稳定性与可读性

---

## 13. 最终结论

本次前台重构采用 **Lite IDE 工作台** 作为总体方案：

- 首页与默认主任务：聊天与模型切换
- 总体结构：一级导航 + 左主侧栏 + 主内容区 + 右侧抽屉
- 工程能力：上下文抽屉化
- 知识库/媒体：首页卡片入口 + 完整独立页
- 设置：左目录 + 顶部搜索的现代设置中心
- 视觉语言：极简浅色生产力风格
- Win7 策略：结构统一、视觉降级

这套方案能在不动核心业务能力闭环的前提下，显著提升 EVA 的现代感、层级清晰度与产品完成度，并且是现有代码结构下可执行、可分阶段落地的前台重构方向。
